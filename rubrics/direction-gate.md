# Rubric：方向门禁

当任务可能影响底层产品逻辑时，在详细方案设计或 PRD 生成前使用这个 Rubric。

## 触发问题

只要任一问题答案为“是”，就应该执行方向门禁：

- 是否新增或改变对象模型？
- 是否改变归属、可见范围或访问边界？
- 是否改变生命周期？
- 是否新增或改变状态迁移？
- 是否改变核心用户路径？
- 是否依赖隐式系统补偿，让方案表面看起来简单？
- 是否把已有能力扩展到新对象、新角色或新入口？

## 检查项

| 检查项 | 通过标准 |
|---|---|
| 问题清晰度 | 真实问题和用户提出的方案已经分开表达 |
| 对象模型 | 关键对象和对象关系已经明确 |
| 归属边界 | 所有者、查看者、编辑者和系统职责已经明确 |
| 生命周期 | 创建、更新、归档、删除、恢复、终态在相关场景下已考虑 |
| 状态机 | 新增或变化的状态与迁移已经列出 |
| 权限 | 访问和可见范围没有被一句“沿用现有逻辑”带过 |
| 现有行为 | 已有产品行为已验证，或明确标记未知 |
| 隐式补偿 | 方案不依赖未说明的自动迁移、复制、恢复或特殊兜底 |
| 范围 | 本期范围和不做范围已经明确 |
| 回退 | 如果方向不稳，建议先回退，而不是继续写 PRD 细节 |

## 输出

- 方向风险：低 / 中 / 高
- 阻塞问题
- 需要补充的证据
- 建议：继续 / 细化方向 / 回退
