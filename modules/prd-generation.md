# 模块：PRD 生成

## 目的

把已确认的产品决策转成可交付、可实现、可测试的产品需求文档。

## 何时激活

- 方案方向已经确认。
- 关键产品逻辑和范围已经清楚。
- 剩余未知项可以被明确列出。

## 执行步骤

1. 复述背景和目标。
2. 定义范围和非目标。
3. 描述用户和场景。
4. 编写功能需求。
5. 编写产品逻辑、状态和边界条件。
6. 补充验收标准。
7. 列出依赖、风险和待确认问题。

## 输出

使用：

```text
templates/prd.md
```

PRD 必须区分：

- 已确认事实
- 产品决策
- 假设
- 待确认问题
- 不做范围

## PRD 输出要求

- 不机械填满模板；与本期无关、没有有效信息或不会影响评审判断的模块允许为空。
- 表达简洁清晰，优先结构化呈现。
- 不重复表达同一逻辑，不在多个位置重复写相同信息。

## 文档配图与平台写回

仅当图示能明显降低理解成本，且交付目标包含在线文档写回时启用。结构简单或文字更清楚的内容不强行配图。

1. 先确定每张图的核心结论、阅读对象和对应正文，再选图；多个独立主题拆成多张聚焦图，不塞进一张大图。
2. 按文档语义自由选择结构图、关系图、分层图、对比图、时序图或信息示意图，不固定套用单一图形模板。
3. 视觉表达应贴近目标文档平台的原生风格：清晰、疏朗、美观，信息层次明确，留白合理，视觉元素和配色统一，不以装饰性压过信息传达。
4. 优先避免把 Mermaid 流程图当作通用解法。只有内容强依赖流程顺序、分支或循环，且没有更合适的替代表达时，才使用 Mermaid 流程图。
5. 图与 PRD 已确认的事实、对象、状态和本期边界必须一致；不得为了视觉完整新增产品规则。
6. 使用目标文档平台的原生文档和画板能力执行定位、插入、更新与预览。若目标为飞书/Lark 且环境提供 `lark-doc` 与 `lark-whiteboard`，按对应 Skill 的完整 Workflow 执行；其他平台使用等价的原生能力。
7. 图示插入直接相关的正文附近，不集中堆在文档末尾，不改动已确认的文档结构。
8. 写回后必须检查平台中的实际预览，并回读文档确认图示位置和正文结构。空白画板、文字溢出、元素重叠、关键连线不清或图义与正文不一致，均视为未完成。
