# 模块：PRD 生成

## 目的

把已确认的产品决策转成可交付、可实现、可测试的产品需求文档。

## 何时激活

- 方案方向已经确认。
- 关键产品逻辑和范围已经清楚。
- 剩余未知项可以被明确列出。

## 执行步骤

1. 复述背景和目标。
2. 定义范围和非目标。
3. 描述用户和场景。
4. 编写功能需求。
5. 编写产品逻辑、状态和边界条件。
6. 补充验收标准。
7. 列出依赖、风险和待确认问题。

## 输出

使用：

```text
templates/prd.md
```

PRD 必须区分：

- 已确认事实
- 产品决策
- 假设
- 待确认问题
- 不做范围

## PRD 输出要求

- 不机械填满模板；与本期无关、没有有效信息或不会影响评审判断的模块允许为空。
- 表达简洁清晰，优先结构化呈现。
- 不重复表达同一逻辑，不在多个位置重复写相同信息。

## 文档配图与平台写回

仅当图示能明显降低理解成本，且交付目标包含在线文档写回时启用。结构简单或文字更清楚的内容不强行配图。

1. 先按 `templates/diagram-brief.md` 确定每张图的核心结论、信息层级、颜色/连线语义和交付形式；多个独立主题拆成多张聚焦图，不塞进一张大图。简单图可在执行上下文中完成简报。
2. 按文档语义自由选择结构图、关系图、分层图、对比图、时序图或信息示意图，不固定套用单一图形模板。
3. 复杂架构图、数据流图、复合流程图和产品方案信息图优先调用 `fireworks-tech-graph` 或等价高保真 SVG/PNG Skill，生成源图和 PNG 预览；只有实际调用并产出可审查文件，才算工具已触发。
4. 需要平台内继续编辑时，以通过审查的 SVG/PNG 为视觉基线，再通过平台原生画板导入或重建；若目标为飞书/Lark，使用 `lark-whiteboard`。转换后必须重新导出平台预览。
5. 优先避免把 Mermaid 流程图当作通用解法。只有内容强依赖流程顺序、分支或循环，且没有更合适的替代表达时，才使用 Mermaid；复杂卡片、多个语义区域或高保真要求不得静默降级为默认 Mermaid。
6. 高保真 Skill 不可用时，说明能力缺口，按用户门禁选择安装、使用等价工具或接受降级，不生成低质量占位图冒充正式交付。
7. 图与 PRD 已确认的事实、对象、状态和本期边界必须一致；不得为了视觉完整新增产品规则。
8. 高保真图示生成后先执行 `rubrics/diagram-visual-quality.md`；通过后再使用目标文档平台的原生文档能力定位和插入。飞书/Lark 使用 `lark-doc`，其他平台使用等价能力。
9. 图示插入直接相关的正文附近，不集中堆在文档末尾，不改动已确认的文档结构。
10. 写回后检查平台实际预览并回读文档。只通过语法、重叠或溢出检查，没有查看 PNG 和平台视觉效果，不视为完成。
