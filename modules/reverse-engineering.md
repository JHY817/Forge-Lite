# 模块：逆向工程

## 目的

通过代码或实现事实理解现有产品逻辑和设计约束。

## 何时激活

- 需求可能影响权限、生命周期、状态迁移、数据流、删除恢复或跨模块行为。
- 文档不完整或互相冲突。
- 方案是否可行依赖实现事实。

## 输入

- 已配置的代码库路径
- 相关产品行为
- 搜索线索
- 已知 API、路由、模型或服务

## 执行步骤

1. 定位相关入口。
2. 追踪核心逻辑和状态变化。
3. 识别权限、数据和依赖边界。
4. 区分代码事实和产品决策。
5. 记录未知项和高风险假设。
6. 总结设计约束。
7. 使用 `rubrics/reverse-engineering-quality.md` 检查搜索范围、执行链路、证据强度、多服务覆盖和不确定性。

## 输出

- 代码事实
- 影响链路
- 现有状态和迁移
- 权限和数据边界
- 实现约束
- 未知项
- 对产品设计的建议影响

## 事实边界

- 使用 `workflows/context-and-evidence.md` 的事实类型。
- 代码事实不自动等于线上产品事实。
- 单个仓库或服务未发现能力，不能证明产品不存在该能力。
- 事实 Rubric 不通过时，不能因为文档表达清晰就进入方案阶段。
