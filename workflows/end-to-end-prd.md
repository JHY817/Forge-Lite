# 端到端 PRD 工作流

当用户希望从需求输入推进到方案判断和 PRD 时，使用这个流程。

## 目标

不要在需求刚出现时直接写 PRD。  
只有在上下文、证据和方案方向足够明确后，才进入 PRD 生成。

## 流程

```text
1. 判断任务类型
2. 需求理解
3. 产品现状检查
4. 证据计划
5. 方向门禁
6. 按需执行其他模块
7. 方案设计
8. 用户确认
9. 按需做原型设计准备
10. PRD 生成
11. 审计与经验沉淀
```

## 起手输出

复杂任务开始前，先输出设计计划：

- 任务等级：小 / 中 / 大
- 核心问题
- 建议执行的模块
- 建议跳过的模块及原因
- 需要补充的证据
- 预期产物
- 用户确认节点
- 已知风险和未知问题

## 模块选择

只运行必要模块：

- 默认先考虑 `demand-understanding`。
- 如果依赖已有产品行为，运行 `product-current-state`。
- 如果涉及代码事实、权限、生命周期、状态或数据链路，运行 `reverse-engineering`。
- 如果优先级或影响范围需要数据支持，运行 `data-analysis-decision`。
- 如果外部产品能帮助判断方向，运行 `competitor-analysis`。
- 只有方案方向确认后，才运行 `prototype-preparation`。
- 只有方案方向确认后，才运行 `prd-generation`。

## 完成标准

流程完成时应满足：

- PRD 基于已验证事实，或明确标注了未知项。
- 方向级风险已经检查。
- 产品特定缺口已经列出。
- 审计结果已经通过，或转成明确返工项。
- 可复用经验只提出建议，不静默写入长期沉淀。
