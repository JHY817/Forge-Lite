# 端到端 PRD 工作流

当用户希望从需求输入推进到方案判断和 PRD 时，使用这个流程。

## 目标

不要在需求刚出现时直接写 PRD。  
只有在上下文、证据和方案方向足够明确后，才进入 PRD 生成。

## 流程

```text
1. 判断任务类型
2. 读取或创建当前状态
3. 按上下文与证据协议装配阶段输入
4. 需求理解与产品现状检查
5. 证据计划
6. 方向门禁
7. 按需执行其他模块
8. 方案设计
9. 按 auto / notify / approve 执行门禁
10. 按需做原型设计准备
11. 原型工具执行后由 Agent 验收输出
12. PRD 生成
13. 审计、状态迁移与经验沉淀
```

## 起手输出

复杂任务开始前，先输出设计计划：

- 工作量等级：小 / 中 / 大
- 决策风险：low / medium / high
- 核心问题
- 建议执行的模块
- 建议跳过的模块及原因
- 需要补充的证据
- 预期产物
- 各节点门禁：auto / notify / approve
- 已知风险和未知问题

## 模块选择

只运行必要模块：

- 默认先考虑 `demand-understanding`。
- 如果依赖已有产品行为，运行 `product-current-state`。
- 如果涉及代码事实、权限、生命周期、状态或数据链路，运行 `reverse-engineering`。
- 如果优先级或影响范围需要数据支持，运行 `data-analysis-decision`。
- 如果外部产品能帮助判断方向，运行 `competitor-analysis`。
- 只有方案方向确认后，才运行 `prototype-preparation`。
- 只有方案方向确认后，才运行 `prd-generation`。

工作量等级只控制执行深度，决策风险控制证据强度、方向门禁和用户批准。复杂但事实完整的任务不必机械询问用户；简单任务触碰对象模型时仍可能是高风险。

逆向工程完成后先执行 `rubrics/reverse-engineering-quality.md`。原型输出进入用户确认或 PRD 前执行 `rubrics/prototype-output-quality.md`。

## 完成标准

流程完成时应满足：

- PRD 基于已验证事实，或明确标注了未知项。
- 方向级风险已经检查。
- 产品特定缺口已经列出。
- 审计结果已经通过，或转成明确返工项。
- 阶段状态、失效产物和下一步已经更新。
- 可复用经验只提出建议，不静默写入长期沉淀。
