# 用户门禁与受控并行协议

## 三类门禁

| 类型 | Agent 行为 | 适用情况 |
|---|---|---|
| `auto` | 自主执行，不中断用户；阶段结束记录 | 范围内、低风险、可逆、已有规则可判定 |
| `notify` | 执行安全动作后同步结论和风险 | 中风险但可逆；P2；不改变已确认边界 |
| `approve` | 停止在决策前等待明确确认 | 改变范围/对象模型；接受 P1；不可逆、高成本或外部高风险动作 |

用户没有回复不等于批准；用户已在当前任务中授权的范围不重复申请。

## 必须 approve

- 改变已确认范围；
- 改变核心对象、归属、权限、生命周期或主路径；
- 接受 P1 风险；
- 删除、批量覆盖、权限变更、对外发布或发送消息；
- 明显扩大研究成本；
- 多个合理方向依赖用户未公开的决策权重。

P0 不能以“用户接受”为由标记质量通过。

## 受控并行

只在大型任务且子问题能独立取证时启用。例如按服务逆向、按证据源研究、推荐方向与反方风险独立分析。

要求：

1. 每个子任务有明确问题、范围、证据和禁止扩展项；
2. 子任务只提交证据和局部判断，不直接修改主结论；
3. 主 Agent 负责冲突判断和最终方案；
4. 多数意见不能替代事实；
5. 小任务、高度耦合任务和同一正式文档改写保持单线程。
