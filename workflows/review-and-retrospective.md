# 审计与复盘流程

当需要审计 PRD、方案、原型设计 Brief 或阶段产物时，使用这个流程。

## 目标

识别产品风险、缺失证据、方向问题，以及是否需要返工或回退。

## 流程

```text
1. 判断产物类型
2. 选择相关 Rubric
3. 按 `rubrics/issue-severity.md` 审计
4. 建立稳定问题 ID 和问题账本
5. 判断增量复审 / 全量复审 / 回退
6. 更新阶段状态和失效产物
7. 提出经验沉淀与 Eval 建议
```

## 审计优先级

先讲问题，不要先写表扬。

优先检查：

1. 方向级问题。
2. 缺失或无法验证的事实。
3. 对象模型、生命周期、权限、状态机问题。
4. 依赖系统隐式补偿的规则。
5. 实现边界不清的问题。
6. 缺失验收标准的问题。
7. 表达和结构问题。

## 输出格式

- 本轮摘要：阶段、产物、Rubric、P0/P1/P2 数量和下一步。
- 问题账本：`issue_id`、严重度、状态、位置、证据、根因阶段、影响、返工动作、复审范围和门禁。
- 复审结论：修复、未修复、重开、新增和回归问题。
- 通过 / 细化 / 回退建议。
- 可选经验沉淀和 Eval 建议。

P0 必须修复或回退；P1 需要 approve；仅有 P2 时可以 notify 后推进。范围、对象、归属、权限、生命周期、核心路径或关键事实变化时，下游全量重审。

长期经验沉淀必须经过用户确认。
