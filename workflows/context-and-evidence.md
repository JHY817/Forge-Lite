# 上下文与证据协议

> 控制每个阶段加载的信息，并区分事实、推断、假设和冲突。

## 阶段上下文包

进入模块时只装配：

1. 当前状态；
2. 本阶段目标和退出条件；
3. 与本阶段有关的已确认事实；
4. 当前适用规则；
5. 上一阶段交接摘要；
6. 本阶段输出契约。

不要因为目录存在就加载所有历史文档、所有 Rubric、全部代码仓或全部经验记录。

## 事实类型

| 类型 | 含义 | 使用边界 |
|---|---|---|
| `confirmed_product_fact` | 经产品、可信文档或多来源验证的产品行为 | 可支持方案 |
| `confirmed_code_fact` | 经真实执行链路确认的实现行为 | 不自动等同线上表现 |
| `confirmed_data_fact` | 口径和范围明确的数据结论 | 仅在口径内使用 |
| `user_statement` | 用户提供的业务或组织信息 | 与可查事实冲突时需标记 |
| `inference` | 从事实推导出的解释 | 不能伪装成事实 |
| `assumption` | 为继续执行采用的临时前提 | 仅低风险、可逆时可继续 |
| `conflict` | 可信来源不一致 | 不强行调和 |
| `unverified` | 尚未查证 | 不得决定高风险方向 |

关键事实建议使用 `fact_id`，供方案、PRD 和审计引用。

## 查证顺序

1. 当前状态和主产物；
2. 已配置的产品知识和历史决策；
3. 用户提供的原始材料和数据；
4. 代码、配置、日志或实际运行表现；
5. 无法从其他来源得到的隐性业务信息。

能查的事实不问用户。单个仓库或文件未发现某能力，只能得出“当前搜索范围未发现”，不能证明产品不存在该能力。

## 信息对齐的触发条件

仅当以下条件同时满足时暂停并询问用户：

1. 信息无法从已配置来源查证；
2. 不同答案会改变范围、对象模型、核心路径或重大取舍；
3. 默认假设可能造成高风险、不可逆后果或明显返工。

否则使用安全、可逆的合理假设继续，并明确标注。

## 阶段结束时的压缩

只传递已确认事实及证据、决策及理由、设计约束、冲突与假设、失效产物、Rubric 结果和下一步。搜索过程、大段代码和重复规则只保留引用。
