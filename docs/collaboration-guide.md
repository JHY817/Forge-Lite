# 协作说明

这份文档说明人应该怎么和 FORGE Lite 协作。

## 怎么开始一个任务

最好提供这些信息：

- 你想解决的问题。
- 目标用户是谁。
- 影响哪个对象、入口或流程。
- 你期望的产物是什么。
- 已有材料在哪里。
- 有哪些已知约束。
- 有没有明确不做的范围。

不需要一开始就准备得很完整。  
FORGE Lite 应该先判断缺什么，再追问关键缺口。

## 好的输入示例

```text
我们想把一个已有能力开放给新的用户角色。
请帮我判断现有逻辑能不能复用，需要补哪些事实，
以及这个需求是否应该进入 PRD。
```

这个输入比较好，因为它说明了：

- 是已有能力扩展。
- 涉及新用户角色。
- 需要判断复用和事实补证。
- 不要求 Agent 直接写 PRD。

## 不太好的输入示例

```text
帮我写个 PRD。
```

这个输入不是不能处理，但如果任务复杂，FORGE Lite 不应该直接写。  
它应该先判断：

- 需求是否已经收敛。
- 产品现状是否清楚。
- 是否触碰对象模型、权限、生命周期或状态机。
- 是否需要先做方案判断。

## Agent 应该做什么

FORGE Lite 应该：

- 先判断任务阶段。
- 输出设计计划。
- 只激活必要模块。
- 能查事实就查事实。
- 区分事实、判断、假设和待确认问题。
- 遇到方向问题时先停下来审方向。
- 必要时建议回退，而不是继续补细节。

## 人应该做什么

人应该：

- 提供业务背景。
- 确认设计计划。
- 确认关键方向。
- 提供必要材料。
- 判断哪些风险可以接受。
- 决定是否进入 PRD。
- 决定是否沉淀经验。

## 关键确认节点

常见确认节点：

1. 执行计划确认。
2. 需求边界确认。
3. 方向判断确认。
4. 原型设计输入确认。
5. PRD 初稿确认。
6. 审计返工确认。
7. 经验沉淀确认。

## 一个简单协作过程

```text
用户输入粗略需求
→ Agent 判断任务阶段
→ Agent 输出设计计划
→ 用户确认范围
→ Agent 做必要事实补证
→ Agent 给出方案判断
→ 用户确认方向
→ Agent 生成 PRD 或原型准备材料
→ Agent 审计产物
→ 用户确认是否沉淀经验
```

FORGE Lite 的价值不在于“更快写文档”，而在于减少漏判、错判和无效返工。
