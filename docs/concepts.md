# 核心概念

## 产品经理工作流 Agent

产品经理工作流 Agent 不是单纯写文档的工具。

它应该帮助产品任务在正确阶段推进：

- 需求理解
- 产品现状理解
- 事实补证
- 方案设计
- 原型准备
- PRD 生成
- 审计返工
- 经验沉淀
- Eval 与回归验证

## 先判阶段，再做产物

很多产品任务失败，不是因为文档写得慢，而是因为太早进入产物。

FORGE Lite 的第一原则是：

```text
先判断任务处在哪个阶段，再决定产出什么。
```

比如：

- 需求不清楚，就先做需求理解。
- 现状不清楚，就先查产品现状。
- 方向不稳，就先做方向门禁。
- 方案已确认，再写 PRD。

## 方向门禁

方向门禁用于检查产品方向是否稳。

只要任务触碰这些内容，就要先审方向：

- 对象模型
- 归属边界
- 生命周期
- 权限
- 状态机
- 核心路径
- 系统隐式补偿

如果方向不稳，继续写 PRD 只会制造更多返工。

## 运行状态

阶段状态使用统一协议保存。Agent 不应从多份历史文档猜测“哪一版有效”，方向回退后也必须明确哪些下游产物失效。

见：`workflows/runtime-state.md`。

## 事实与推断

产品事实、代码事实、数据事实、用户陈述、推断、假设、冲突和未验证信息必须区分。大上下文不能替代证据质量，单一来源未发现也不能证明能力不存在。

见：`workflows/context-and-evidence.md`。

## 用户门禁

FORGE Lite 默认人仍然负责关键决策。

但不是每个阶段都要中断用户：

- `auto`：范围内、低风险、可逆，Agent 自主推进。
- `notify`：先执行安全动作，再同步结论和风险。
- `approve`：改变范围、对象模型、接受 P1 风险或执行不可逆动作前确认。

见：`workflows/approval-and-parallelism.md`。

## Eval

规则写得更完整不等于 Agent 更可靠。流程和 Rubric 调整需要用 `evals/` 中的通用案例或使用者自己的私有案例验证。未验证的调整应明确标记“待验证”。

## 本地上下文

FORGE Lite 本身是通用框架。

真正让它有用的是你的本地上下文：

- 产品背景
- 知识库
- 代码库
- 业务规则
- 团队模板
- 私有经验沉淀

这些内容应该由使用者自己配置，不应该内置在开源仓库里。
