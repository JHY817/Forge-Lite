# GitHub 开源注意事项

这份清单是给第一次做 GitHub 开源项目的人看的。

## 发布前必须检查

发布到 GitHub 前，先检查这些内容有没有误提交：

- 公司名、产品名、客户名。
- 内部路径、本机路径、私有代码库地址。
- 真实 PRD、真实客户需求、真实项目复盘。
- 内部截图、带数据的图片、用户信息。
- API key、token、cookie、密码、`.env` 文件。
- 私有知识库内容。
- 私有经验沉淀。
- 业务专属 Rubric。

## 建议先建私有仓库

第一次开源不要一上来就建 public 仓库。

建议流程：

```text
本地整理
→ 创建 GitHub private repo
→ push 到 private repo
→ 在 GitHub 页面再检查一遍
→ 选择 License
→ 确认没有敏感信息
→ 再切换成 public
```

这样比较稳。

## License 很重要

开源前一定要选 License。

如果没有 License，别人其实不能明确知道是否可以合法使用、修改、分发你的项目。

常见选择：

| License | 适合场景 |
|---|---|
| MIT | 简单、宽松，适合大多数工具和框架 |
| Apache-2.0 | 也很宽松，比 MIT 更正式，包含专利授权条款 |
| CC BY 4.0 | 更适合纯文档，不太适合代码或工具框架 |

我对 FORGE Lite 的初步建议：

- 如果你想降低使用门槛：选 MIT。
- 如果你想更正式一点：选 Apache-2.0。

## README 要讲清楚边界

README 里要明确说明：

- 这是通用框架。
- 不包含任何公司或产品专属知识。
- 使用者需要配置自己的产品背景、知识库、代码库和 Rubric。
- 不要提交真实业务案例或私有内容。

## `.gitignore` 要保护私有配置

当前已经建议忽略：

```text
config/
private/
learning/
.env
.env.*
```

后续真实使用时：

- `config.example/` 可以提交。
- `config/` 不要提交。
- `learning/` 不要提交。
- 私有案例不要提交。

## 示例要虚构

所有开源示例都建议用虚构产品。

不要用：

- 真实功能名。
- 真实业务背景。
- 真实用户反馈。
- 真实截图。
- 真实数据。

可以用：

- 虚构产品。
- 公开产品。
- 抽象场景。

## 第一个版本不要太大

第一版开源项目越大，越难维护，也越容易泄露边界。

建议第一版只包括：

- README
- Agent 模板
- 配置示例
- 核心工作流
- 核心模块
- 通用 Rubric
- 通用模板
- 一个虚构示例

高级模块和英文版可以后续慢慢补。

## 维护时要注意

如果以后有人提 issue 或 PR：

- 不要让别人贴自己的公司机密。
- 不要接受真实客户案例。
- 不要接受带私有业务逻辑的 Rubric。
- 鼓励用虚构例子讨论问题。
- 把框架问题和具体业务问题分开。

## 发布前搜索建议

发布前在本地搜索：

```text
公司名
产品名
客户名
/Users/
password
token
secret
api_key
.env
```

搜不到敏感内容，再考虑公开。
